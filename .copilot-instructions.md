# Copilot Instructions - Brain Index AI

## Decision Making

- **Complex Issues**: Assess all scenarios thoroughly. Choose the solution that best fits the use case, prioritizing maintainability and scalability.
- **Trade-offs**: Favor modern patterns over legacy approaches. If uncertain, ask clarifying questions.

## Code Standards

### Modern Conventions

- **Next.js 16**: Use App Router, Server Components by default, Client Components only when needed (`'use client'`)
- **Supabase**: Prefer Server Components for auth checks, use RLS policies, leverage JSONB for flexible data
- **Shadcn UI**: Use existing components, maintain Tailwind CSS v4 patterns, follow Radix UI composition
- **TypeScript**: Strict mode, explicit types for function params/returns, avoid `any`

### File Header Comments

Every new file must start with:

```typescript
/**
 * Purpose: [What this file does in 1 sentence]
 * Key Parts: [Main functions/exports - 2-3 items]
 * Used By: [What uses this / Where it's imported]
 * Why: [Why this exists in the architecture]
 */
```

### Function Comments

```typescript
// Fetch article content from URL and extract main text using Readability
// 1. Fetch raw HTML with timeout handling
// 2. Parse DOM with jsdom
// 3. Extract article content via Readability algorithm
// Returns: sanitized HTML + metadata (title, author, reading time)
async function parseArticle(url: string) { ... }
```

- **One-liner**: Purpose of the function
- **Steps**: Numbered list of main operations (2-5 steps max)
- **Returns**: What it returns and key properties
- **No bloat**: Skip obvious comments, focus on "why" not "what"

## Architecture Patterns

### API Routes

- **Naming**: Resource-based (`/api/articles`, `/api/tags`)
- **Auth**: Check session at route start, return 401 early
- **Errors**: Return structured JSON with status codes
- **Logging**: Log all requests to `api_logs` table for cost tracking

### Database

- **RLS**: Every table must have RLS policies (user isolation)
- **Constraints**: Use database constraints (UNIQUE, NOT NULL) over app-level validation
- **Migrations**: Never modify schema directly, always create migrations

### Components

- **Server First**: Default to Server Components, only use Client Components for interactivity
- **Composition**: Small, focused components over large monoliths
- **Props**: Explicit interfaces, no prop drilling beyond 2 levels

### Styling (Tailwind + Shadcn)

- **Tailwind v4**: Use modern syntax, leverage CSS variables for theming
- **Utility-First**: Apply Tailwind classes directly, avoid custom CSS unless necessary
- **Spacing**: Use consistent scale (`gap-4`, `p-6`, `space-y-2`) - prefer `gap` over `space-*` in flex/grid
- **Responsive**: Mobile-first (`sm:`, `md:`, `lg:`), design for small screens first
- **Dark Mode**: Use `dark:` variants, rely on CSS variables from Shadcn theme
- **Colors**: Use semantic tokens (`bg-background`, `text-foreground`, `border`) over hard-coded colors
- **Shadcn Components**: Don't modify base components, extend with wrapper components if customization needed
- **Consistency**: Match existing UI patterns (buttons, inputs, cards) before creating new ones
- **Accessibility**: Include `aria-*` labels, focus states (`focus-visible:ring-2`), keyboard navigation
- **Class Order**: Group logically (layout → spacing → colors → typography → effects)

```tsx
// Good: Semantic, responsive, accessible
<Button
  variant="outline"
  size="sm"
  className="w-full sm:w-auto gap-2 focus-visible:ring-2"
  aria-label="Save article"
>
  <SaveIcon className="h-4 w-4" />
  Save
</Button>

// Bad: Hard-coded colors, no responsiveness
<button className="bg-blue-500 text-white px-4 py-2">
  Save
</button>
```

## Performance

- **Optimize Early**: Consider pagination, lazy loading, memoization from the start
- **Caching**: Cache AI results permanently, cache article parsing for 24h
- **Bundle Size**: Tree-shake imports (`import { x } from 'lib/utils'` not `import * as utils`)

## Security

- **Env Vars**: Never commit secrets, use `.env` and Vercel env vars
- **Input Validation**: Sanitize all user inputs, validate on both client and server
- **RLS Policies**: Double-check all queries use proper user filtering

## Efficiency Rules

1. **DRY**: Extract repeated logic into utilities
2. **Single Responsibility**: One function/file does one thing well
3. **Early Returns**: Guard clauses at function start
4. **Async/Await**: Prefer over callbacks/promises chains
5. **Error Boundaries**: Wrap risky operations, provide fallbacks

## Version Control

- **Commit Important Changes**: After completing meaningful work (feature, fix, refactor), commit changes
- **Quality Messages**: Use clear, descriptive commit messages following conventional commits format:
  - `feat: add article parsing service with Readability`
  - `fix: resolve OAuth redirect loop in auth flow`
  - `refactor: simplify database schema to 3 tables`
  - `docs: update implementation plan with citation system`
- **Confirm Before Committing**: Always ask user for confirmation before committing, summarizing what was accomplished
- **Logical Grouping**: Commit related changes together (don't mix unrelated features)
- **Track Progress**: Reference phase numbers from IMPLEMENTATION_PLAN.md in commits when applicable
